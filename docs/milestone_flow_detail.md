# Aegis QA Assistant — 동작 방식 상세 플로우 (중간 마일스톤 보고용)

> 작성일: 2026-07-03
> 목적: 임원 보고용 시스템 동작 방식 세부 기술

---

## 전체 개요

Aegis QA Assistant는 **기획서(PDF) 한 장**을 업로드하면 AI가 자동으로 흐름 트리(Flow Tree)를 만들고, 그 트리를 기반으로 테스트케이스(TC)를 생성해 QA 팀이 바로 쓸 수 있는 Excel 파일을 내보내는 시스템입니다.

```
기획서 PDF 업로드
    ↓
AI가 화면 구조 파악 (구조적 트리)
    ↓
AI가 사용자 인터랙션 흐름 생성 (흐름 트리)  ← 핵심 산출물
    ↓
AI가 흐름 트리를 자동 점검 + 자동 교정
    ↓
흐름 트리 + TC가 나란히 붙은 Excel 1장 다운로드
```

## 1단계: 프로젝트 등록

### 1-1. 프로젝트 생성

**사용자 동작:**
- 메인 화면(/) 진입 → 프로젝트 목록 테이블 확인
- "+ 새 프로젝트" 버튼 클릭 → 모달/폼 열림
- 필수 입력: 프로젝트명, (선택) 설명, 서비스 URL, QA 룰셋 선택

**QA 룰셋 선택 시 의미:**
- 기본값: "웹 서비스 공통 (XPERP)" — 시스템이 자동 생성한 표준 룰셋
- 사용자가 별도 룰셋 생성 후 선택 가능 (예: "XpBIZ 전용", "모바일앱 전용")
- 선택한 룰셋은 이 프로젝트의 **모든 기획서 분석·TC 생성에 일관 적용**됨

**백엔드 처리:**
```
POST /api/projects/
  → projects 테이블 INSERT (name, description, ruleset_id)
  → 생성된 project_id 반환
```

## 2단계: 기획서 업로드

### 2-1. PDF 업로드 준비

**지원 포맷:**
- Excel → PDF 변환본 (텍스트 기반): 브레드크럼, 버튼명, 메시지 텍스트가 추출 가능
- Figma → PDF 변환본 (이미지 기반): Gemini Vision API로 이미지 직접 분석
- 시스템이 두 포맷을 **자동 감지** 후 처리 방식 분기

**사용자 동작:**
- 프로젝트 클릭 → 문서 목록(/projects/{id})
- TC 생성 레벨 선택: [1][2][3][4][5] (기본값: Lv.3 정밀 검증)
  - Lv.1: 핵심 정상 흐름 + 대표 실패 케이스만
  - Lv.3: 업무 규칙·권한 분기·상태 전이까지 (기본값, 약 200~500 TC)
  - Lv.5: 기획서·매뉴얼·결함이력에서 도출 가능한 전수 케이스
- PDF 파일 선택 → "업로드" 클릭

**백엔드 처리:**
```
POST /api/documents/{project_id}/upload  (multipart/form-data)
  → uploads/ 폴더에 UUID 접두사로 파일 저장
  → documents 테이블 INSERT (filename, original_filename, file_path, tc_level)
  → status = 'uploaded'
  → 백그라운드 분석 작업 즉시 시작 (FastAPI BackgroundTasks)
```

## 3단계: 구조적 메뉴트리 자동 추출 (백그라운드)

### 3-1. PDF 타입 감지

```python
is_text_based_pdf(pdf_path)
  → PDF 내 텍스트 레이어 존재 여부 체크
  → True: pypdf로 텍스트 추출 후 Gemini에 텍스트로 전달
  → False: pdf_to_base64_chunks()로 이미지화 후 Gemini Vision으로 전달
```

### 3-2. 구조적 트리 추출 (Gemini API 1차 호출)

**목적:** 기획서에 어떤 화면들이 있고 어떻게 계층화되는지 파악

**Gemini에 전달하는 정보:**
- PDF 전체 내용 (텍스트 or 이미지)
- 원본 파일명 (보조 단서)
- TREE_PROMPT: "모듈 > 화면 > 영역 > 기능" 4단계 계층 추출 지시
  - **1순위**: 기획서 본문의 "A > B > C" breadcrumb (예: "XpERP > 수납 > 미납조회 > 미납대장")
  - **2순위**: 본문의 화면 제목·섹션 헤더
  - **3순위**: 파일명/표지 (가장 약한 단서, 기획서 내용이 있으면 무시)

**Gemini 출력 (구조적 트리 JSON):**
```json
{
  "title": "간편전자고지서 신청 QR 탑재 안내문 출력 기획서",
  "tree": [
    {
      "id": "1",
      "name": "수납",
      "children": [
        {
          "id": "1-1",
          "name": "미납조회",
          "children": [
            {
              "id": "1-1-1",
              "name": "미납대장",
              "children": [
                {
                  "id": "1-1-1-1",
                  "name": "간편전자고지서 QR 안내문 다운로드 버튼",
                  "spec_page": "5",
                  "key_points": ["1차 오픈 대상 단지 조건", "권한별 알림톡 발송", "PDF 다운로드"],
                  "change_type": "new_feature",
                  "children": []
                }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

**목적:** 동일 화면에서도 상태(권한/계약코드/이력 등)에 따라 결과가 다른 케이스를 체계적으로 파악

## 4단계: 사용자 — 메뉴트리 검토 (TreeViewPage)

**사용자 동작:**
- 문서 목록에서 "analyzed" 상태 문서 클릭 → TreeViewPage(/tree/{document_id})
- AI가 추출한 화면 계층 구조 확인

- 필요 시 구조적 트리 Excel 다운로드 후 내용 확인

**이 단계의 의미:**
- QA 팀이 "우리가 테스트할 화면은 이게 맞나?" 검토
- 잘못 추출된 화면명이나 누락된 기능 발견 시 수정
- 여기서 확정된 구조적 트리가 **흐름 트리 생성의 골격(Scaffold)**이 됨

## 5단계: 흐름 트리 추출 (백그라운드, 2단계 파이프라인)

### 5-1. 사용자 흐름 트리 추출 시작

**사용자 동작:**
- TreeViewPage에서 "흐름 트리 추출" 버튼 클릭
- 화면에 경과 시간 타이머 표시 ("흐름 트리 추출 중... 1분 30초 경과")
- 완료 시 화면이 자동으로 흐름 트리 탭으로 전환

**백엔드 트리거:**
```
POST /api/documents/{id}/flow-tree
  → doc.flow_tree = None  (즉시, 이전 트리 무효화)
  → db.commit()
  → BackgroundTasks에 _build_flow_tree_bg() 등록
  → 즉시 200 OK 반환 (작업은 백그라운드에서 계속)
```

**프론트엔드 폴링:**
```
매 5초마다 GET /api/documents/{id}/flow-tree
  → ready=False: 아직 생성 중, 계속 폴링
  → ready=True: 완료, flow_tree 데이터 수신 후 화면 갱신
  → error: 실패 메시지 표시
  → 10분 초과 시: "페이지 새로고침 후 확인" 안내
```

### 5-3. 흐름 트리 생성 (Gemini API 3차 호출)

**Gemini에 전달하는 정보:**
1. PDF 전체 내용 (텍스트 or 이미지)
2. FLOW_PROMPT (노드 타입 정의 + 자가 점검 체크리스트)
3. 룰셋 `flow_rules` (정밀도 규칙 10개 + 구조 문법 규칙 13개 + Few-shot 예시)
4. 룰셋 `tree_rules` (관점 가이드)
5. **[화면 골격 — 절대 규칙]**: 위에서 만든 화면 목록
   - "각 화면당 정확히 루트 PR 1개만 만들어라"
   - "같은 화면의 모든 기능은 그 루트의 형제 분기로 통합하라"

**흐름 트리 노드 타입:**

## 6단계: 자동 교정 루프 (백그라운드, 계속)

### 6-1. 커버리지 점검 (Gemini API 4차 호출)

**Gemini에 전달하는 정보:**
- 흐름 트리 전체 텍스트 개요
- 룰셋 규칙 전문 (flow_rules + tree_rules, 섹션 라벨로 구분)
- 점검 프롬프트: "규칙 위반·누락만 찾아라"

**점검 출력 예시:**
```json
{
  "findings": [
    {
      "rule": "팝업/바텀시트 자체를 컨테이너 D 한 줄로 표현 금지",
      "target": "flow",
      "severity": "violation",
      "fixability": "not_followed",
      "where": "알림톡 발송 팝업 → D: 팝업 출력",
      "detail": "팝업 자체를 D 한 줄로 표현. 내부 타이틀·버튼·텍스트를 개별 D로 분해해야 함",
      "suggestion": "D: 팝업 타이틀, D: 발송 버튼, D: 닫기 버튼 등으로 분해"
    },
    {
      "rule": "spec_page 누락",
      "target": "flow",
      "severity": "missing",
      "fixability": "not_followed",
      "where": "전체 노드",
      "detail": "트리의 모든 노드에 spec_page 필드 미입력"
    }
  ]
}
```

**fixability 분류:**
- `not_followed`: 룰셋 규칙이 있고 AI가 안 따른 것 → **자동 교정 대상**
- `spec_limited`: 기획서에 정보 없어서 만들 수 없는 것 → 자동 교정 불가, 기획서 보강 필요

### 6-2. 자동 교정 (Gemini API 5차 호출)

`not_followed` 위반이 1건 이상이면 교정 패스 실행:

**Gemini에 전달하는 정보:**
- 위반 목록 (자연어로 정리된 각 위반 상세)
- 현재 흐름 트리 JSON 전체
- 기획서 PDF 원문 (팝업 내용 분해 등 원문 참조 필요한 교정을 위해)
- REPAIR_PROMPT: "이 위반들만 기획서 원문을 보고 수정해라, 나머지는 그대로"

**교정 결과:**
- 팝업 한 줄 D → 타이틀/텍스트/버튼별 개별 D로 분해
- spec_page 공란 → 기획서 페이지 번호 채움
- PR content에 "진입된 상태" → "진입 직전 상태"로 수정 + C 노드 추가
- 구조 오류 (C가 D의 자식) → C를 D의 형제로 이동

**최종 저장:**
```python
doc.flow_tree = json.dumps(repaired_flow)
doc.status = 'analyzed'   # 흐름 트리는 analyzed 상태 유지
db.commit()
# 콘솔 로그: "[흐름트리 교정 완료] 노드 247개"
```

## 8단계: 핵심 출력 — 흐름트리 + TC 통합 Excel

### 8-1. "📥 흐름트리 + TC Excel" 버튼 클릭

**UX 동작:**
- 버튼 → "⏳ TC 생성 중..."으로 변경 + 비활성화
- 버튼 옆에 경과 시간 실시간 표시 ("TC 생성 중... 2분 14초 경과")
- 완료 시 Excel 파일 자동 다운로드 + 버튼 원상 복귀
- 실패 시 에러 메시지 4초 표시 후 사라짐

**백엔드 처리 3단계:**

**[1단계] 흐름 트리 렌더링 + 경로 수집:**
```python
out_paths = {}
render_flow_tree_excel(flow_tree, _out_paths=out_paths)
# out_paths = {
#   4: [PR노드, C노드, PR노드, D노드, D노드, D노드, D노드],  # row4의 전체 경로
#   5: [PR노드, C노드, PR노드, C노드, D노드],                  # row5의 전체 경로
#   ...
#   77: [PR노드, C노드, V노드, D노드]                          # row77의 전체 경로
# }
```

**각 행(row)의 경로 구성 원리:**
- 흐름 트리는 "루트 PR → C(첫 번째 액션) → D1 D2 D3...Dn (연속 표시 요소)"를 한 행에 표현
- 분기(PR 조건, C 선택지)가 발생하면 새 행으로 분리
- `path_tracker`: 각 행에 렌더링된 모든 노드를 순서대로 누적 (최종 노드만 덮어쓰지 않고 전부 수집)

**[2단계] Gemini가 각 경로를 자연어 TC로 변환:**
```python
generate_tcs_from_flow_paths(out_paths)  # Gemini 6차~ 호출
```

**배치 처리:**
```
77개 경로 → 30개씩 분할 → 3배치 (30+30+17)
600개 경로 → 30개씩 분할 → 20배치

각 배치마다:
  - 30개 경로를 텍스트로 정리 (노드 타입[내용] 형식)
  - FLOW_TC_ENRICH_PROMPT + 경로 목록 → Gemini 호출
  - 응답: {"tcs": [{"row": 4, "title": "...", "preconditions": "...", ...}]}
  - row 번호로 결과 매핑
  - 배치 실패 시 그 배치만 스킵, 나머지 계속 진행
```

**Gemini에 전달하는 내용 (배치 예시):**
```
row=4 (spec_page=4): PR[XpERP > 수납 > 미납조회 메뉴에 진입하기 직전 상태] 
                    → C[미납대장 메뉴 클릭] 
                    → PR[간편전자고지서 1차 오픈 대상 단지이며 팝업 조건 충족] 
                    → D[팝업 타이틀] → D[서브카피] → D[장점1] → D[장점2] → D[장점3] → D[장점4]

row=5 (spec_page=4): PR[XpERP > 수납 > ...] → C[미납대장 메뉴 클릭] 
                    → PR[1차 오픈 대상 단지이며 팝업 조건 충족] 
                    → D[오프라인 안내문 다운로드 버튼] → C[버튼 클릭] → V[PDF 다운로드 확인]
...
```

**Gemini가 생성하는 자연어 TC (예시):**
```json
{
  "row": 5,
  "title": "간편전자고지서 1차 오픈 대상 단지에서 오프라인 안내문 PDF 다운로드 확인",
  "preconditions": "XpERP에 로그인한 상태이며, 현재 단지가 간편전자고지서 1차 오픈 대상 단지이고 팝업 노출 조건이 충족된 상태",
  "steps": "1. XpERP > 수납 > 미납조회 > 미납대장 메뉴로 이동합니다.\n   → 기대 표시: 간편전자고지서 안내문 다운로드 팝업이 자동 출력되어야 한다.\n2. 팝업 내 '오프라인 안내문 다운로드' 버튼을 클릭합니다.\n   → 기대 표시: 해당 단지의 암호화된 코드·단지명·프로그램 종류(XpERP/XpBIZ)·신청 URL이 적용된 QR 이미지가 병합된 안내문(PDF) 다운로드가 진행되어야 한다.",

### 8-2. 완성된 Excel 구조

**시트 레이아웃:**

```

**행별 1:1 대응:**
- Excel 행 4 = 흐름 트리 row4의 인터랙션 경로 = TC-001
- Excel 행 5 = 흐름 트리 row5의 인터랙션 경로 = TC-002
- ...
- Excel 행 80 = 흐름 트리 row80의 인터랙션 경로 = TC-077

**흐름 트리 컬럼 구조 (B열~):**
- B열: 타입 코드 (PR/C/D...) — 좁은 열, 배경색 (PR=노란, C=파랑, D=초록...)

- 두 열 세트가 깊이(Step)마다 반복: B,C = Step0(사전조건), D,E = Step1, F,G = Step2...

**예시 행:**

  ...
  TC_ID_col = "TC-001"
  preconditions_col = "XpERP에 로그인한 상태이며..."
  steps_col = "1. 미납대장 메뉴로 이동\n → 기대 표시: 팝업 자동 출력"

  spec_page_col = "4"

## 9단계: 기존 경로 — 구조적 TC 생성 (선택적)

흐름 트리 외에 기존 구조적 메뉴트리 기반 TC 생성 경로도 유지됨:

### 9-1. TC 생성 시작

**사용자 동작:**
- TreeViewPage에서 "이 흐름트리로 TC 생성 →" (현재 비활성) 또는
- 구조적 트리 탭에서 "TC 생성 시작" 버튼 클릭

**백엔드 처리:**
```
POST /api/documents/{id}/generate-tc
  → status = 'tc_generating'
  → 백그라운드에서 _process_document() 실행
```

### 9-2. 리프 노드 추출 + Gemini TC 생성

**처리 순서:**
```
1. 구조적 트리에서 리프 노드만 추출 (_collect_leaf_nodes)
   → 각 리프 = 독립된 기능(예: "QR 안내문 다운로드 버튼")

2. RAG 컨텍스트 조회 (기능별)
   → 매뉴얼 벡터 DB 검색 (상위 8청크)
   → 결함 이력 벡터 DB 검색 (상위 5건)
   → TC 이력 벡터 DB 검색 (상위 5건)

3. Gemini TC 생성 (기능별, 순차 처리)
   → TC_GENERATE_PROMPT + 상태 매트릭스 + RAG 컨텍스트 + 룰셋 지침
   → Q1~Q4 결정 트리로 유형 분류
   → 상태 조합 × 사용자 액션 = 별도 TC 분리

4. tc_id, category 코드 강제 지정
   → tc.category = 리프 노드 전체 경로 (예: "수납 > 미납조회 > 미납대장 > QR 다운로드 버튼")
   → 노드 ↔ TC 1:N 매핑 보장

5. DB 저장 + TC 이력 벡터 DB 자동 업데이트
```

**구조적 TC 생성 예시 (1개 기능에서 3개 TC 생성):**
```
기능: "알림톡 안내문 발송하러 가기 버튼"

TC-001: 권한 읽기쓰기 상태에서 버튼 클릭 → 알림톡 발송 팝업 출력
TC-002: 권한 읽기만 상태에서 버튼 클릭 → 권한 부족 알럿 출력
TC-003: 권한 사용안함 상태에서 버튼 클릭 → 권한 부족 알럿 출력
```

## 전체 Gemini API 호출 요약

| 순서 | 함수 | 목적 | PDF 포함 | 배치 |
|------|------|------|----------|------|
| 1차 | `build_menu_tree()` | 구조적 트리 추출 | ✅ 전체 | 1회 |
| 2차 | `extract_state_inventory()` | 상태 매트릭스 추출 | ✅ 전체 | 1회 |
| 3차 | `build_flow_tree()` | 흐름 트리 생성 | ✅ 전체 | 1회 |
| 4차 | `check_flow_coverage()` | 자동 커버리지 점검 | ❌ (트리 텍스트만) | 1회 |
| 5차 | `repair_flow_tree()` | 자동 교정 | ✅ 기획서 참조 | 1회 (위반 있을 때만) |
| 6차~ | `generate_tcs_from_flow_paths()` | 자연어 TC 생성 | ❌ (노드 content만) | 30개 배치 분할 |
| (선택) | `generate_tc_from_tree()` | 구조적 TC 생성 | ✅ RAG 검색 결과 | 기능별 순차 |

> **설계 원칙**: PDF를 직접 보는 단계(1~3차)에서 모든 원문 정보를 노드에 담고, 이후 단계는 노드 content를 활용 → 반복 PDF 전송 최소화, 대용량 PDF(70MB+)에서도 동일 구조로 처리 가능.
