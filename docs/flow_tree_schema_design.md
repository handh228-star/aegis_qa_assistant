# 흐름 트리(Flow Tree) 정식 스키마 전환 설계서

> 목적: 현재의 "구조적 트리"(UI 요소 나열)를 **"흐름 트리"(상태→액션→표시결과→검증의 상호작용 설계도)**로 전환한다.
> 근거: PoC 3차(`docs/PoC_FlowTree_doc82_v*.xlsx`)에서 흐름 트리가 사람 레퍼런스와 동급의 구조·상태분기·입도·문구인용으로 추출됨을 확인. 남은 차이는 태깅 관례 수준.
> 상태: 설계 (구현 전). 2026-06-22 작성.

---

## 1. 배경 & 결정

QA팀의 "메뉴트리"는 우리의 얇은 요소 계층이 아니라, **테스트 설계가 완성된 행동 흐름 트리**다. 파일명 `Menutree및TestCase`가 말하듯 그들에겐 **트리 = 완성된 TC 설계**이고, 우리가 TC 단계에서 하던 일을 트리 단계에서 끝낸다.

**결정: 트리의 정의를 "구조"에서 "행동 흐름"으로 바꾼다.** 그 결과:
- 흐름 트리가 **마스터 산출물**이 되고, **TC는 흐름 트리를 선형화(linearize)한 파생물**이 된다.
- 첫날부터 이어진 원칙(트리=앵커, 노드 1:N TC, 상태=분기, spec_page 추적성)이 하나의 모델로 수렴한다.

## 2. 핵심 개념 전환

| | 구조적 트리 (현재) | 흐름 트리 (신규) |
|---|---|---|
| 본질 | "이 화면에 어떤 요소가 있나" | "어떤 상태에서 뭘 하면 뭐가 표시되나" |
| 노드 | 모듈/화면/영역/요소 (name·description·key_points) | 타입 태깅된 흐름 스텝 (PR/C/D/T/H/V…) |
| 상태 | key_points에 압축 | PR 노드로 명시 분기 |
| TC와의 관계 | 트리에서 별도 AI 패스로 생성 | **트리 경로를 선형화한 것이 곧 TC** |
| 깊이 | 4단계 고정 | 흐름 종료까지 가변 (Step 1~30+) |

## 3. 신규 흐름 트리 스키마

```jsonc
{
  "format": "flow",          // 'structural'(구) | 'flow'(신) — 버전 식별
  "title": "기획서 제목",
  "tree": [
    {
      "id": "1",             // 안정적 노드 ID (피드백·TC 추적 키)
      "type": "PR",          // PR|C|D|T|H|V|DC|RC
      "content": "XpERP > 수납 > 미납조회 > 미납대장 진입된 상태",
      "spec_page": "5",      // 출처 페이지 (D/PR에 특히 중요)
      "menu_path": "XpERP > 수납 > 미납조회 > 미납대장",  // 루트 PR에만: 메뉴 위치(검색·그룹핑용)
      "children": [
        { "id": "1-1", "type": "C", "content": "미납대장 선택", "spec_page": "5",
          "children": [
            { "id": "1-1-1", "type": "D", "content": "타이틀 출력 - 입주민이 직접…", "spec_page": "5", "children": [] }
          ]
        }
      ]
    }
  ]
}
```

- **id**: 트리 내 안정적 식별자. 피드백 타겟팅·TC 역추적의 키. (현재 category=경로 매핑을 id 기반으로 승격)
- **type**: §4 태깅 관례.
- **content**: D 노드는 기획서 실제 문구 인용.
- **spec_page**: 노드별 출처 페이지 (기존 spec_page 상속 구조 재사용).
- **menu_path**: 루트 PR에만. 메뉴 위치를 별도 보관해 검색·정렬·기존 메뉴 계층 호환에 사용.

## 4. 태깅 관례 — **사람(골든) 기준을 디폴트로 채택**

PoC 분석 결과 우리와 사람의 실제 커버리지는 동등하나 라벨링 관례가 달라 수치가 어긋났다.
→ **사람 골든 레퍼런스(`human_Aegis_*.xlsx`)의 관례를 디폴트로 확정**한다. QA는 [QA_flowtree_tagging_agreement.md](QA_flowtree_tagging_agreement.md)에서 항목별로 확인·수정만 하면 된다.

### 4.1 타입 코드 (사람 범례 그대로)
| 코드 | 의미 | 디폴트 기준 (사람) |
|---|---|---|
| **PR** | 사전조건/상태 | 메뉴 진입 상태, 권한/코드/설치 등 분기 조건. **상태 값마다 별도 PR (결과 같아도 분리)** |
| **C** | 클릭/선택/터치 | 버튼·링크·체크박스 등 클릭형 액션 |
| **T** | 입력 | **입력 행위 = T** (예: "동 입력"). "입력란/플레이스홀더 출력"은 D |
| **H** | 호버/툴팁 | 마우스 오버로 툴팁 노출 |
| **DC / RC / DD** | 더블클릭 / 우클릭 / Drag Down | 해당 제스처 (사람 범례에 존재) |
| **D** | 표시/결과 | 팝업·토스트·문구·이미지 등 **노출 요소 하나당 D 하나 (최대 분해)** |
| **V** | 검증 포인트 | **수동 확인이 필요한 검증만** (예: 다운로드 PDF 내용 확인). 일반 화면 출력은 D |

### 4.2 디폴트 결정 (사람 기준) — 상세는 합의서 참조
- **분해 입도**: 표시 요소·목록 항목을 **최대한 잘게 개별 D**로 (장점1/2/3/4 각각, 헤드카피·QR코드·단지명 각각).
- **범위**: 이번 기능뿐 아니라 **같은 화면의 기존 형제 요소(조회·인쇄·새로고침 버튼·필터)도 포함**.
- **상태 분리**: 상태 값마다 별도 PR. 결과가 같아도 합치지 않음.
- **메뉴 경로**: 루트 PR content에 "경로 + 진입 상태"를 함께 기술 (별도 `menu_path` 필드도 유지).
- **문구 인용**: D content에 기획서 원문 **그대로 인용**(요약 금지).
- **클릭→결과**: 액션(C/T)과 결과(D)를 **별도 노드**로 분리.
- **산출물 형태**: **흐름 트리가 마스터**, TC는 선형화 파생 (사람의 `Menutree및TestCase` 일체형 관점).
- **완성 기준**: 모든 상태·요소·흐름을 **종단까지 전수**. 개수 할당 없음.

> 위 디폴트는 [QA_flowtree_tagging_agreement.md](QA_flowtree_tagging_agreement.md)에서 QA가 항목별로 OK/수정만 하면 그대로 구현 스펙이 된다.

## 5. 아키텍처 — 흐름 트리 = 마스터, TC = 파생

```
PDF
 └─ (추출) ──> 흐름 트리 [마스터 산출물]  ← 사람 검토·편집 (TreeViewPage)
                 │
                 ├─ (선형화) ──> TC 목록 [파생]  ← 각 루트→종단 경로 1개 = TC 1건
                 │                                 PR→preconditions, C/T→steps.action,
                 │                                 D→steps.expected/expected_result, V→검증
                 │
                 ├─ (렌더) ──> 흐름 트리 Excel (PoC 그리드)
                 └─ (렌더) ──> TC 목록 Excel (기존 리포트)
```

**핵심: TC는 더 이상 별도 AI 패스가 아니라 흐름 트리의 선형화 결과다.**
- 선형화 = 우리가 PoC에서 만든 trie→grid 렌더링의 **역연산**(grid 행 = 경로 = TC).
- 기존 `testcases` 테이블·TC 리포트·검토 UI를 **그대로 재사용** 가능 (입력 소스만 트리 선형화로 교체).
- `spec_page`는 종단 노드에서 상속 (기존 구조 유지), `category`/추적은 노드 `id` 경로로 승격.

이 모델에서 "TC 레벨(Lv.1~5)"은 **선형화 범위/깊이 정책**으로 재해석된다 (미결 #3).

## 6. 컴포넌트별 영향 & 변경

### 6.1 추출 — `tc_generator.build_menu_tree` → 흐름 추출
- `TREE_PROMPT` → `FLOW_PROMPT`로 교체 (PoC v3 프롬프트 기반: 표시요소 개별 D, 형제요소 포함, 상태별 PR, 종단까지 전개, 문구 인용).
- 출력 스키마를 §3으로. JSON 모드 + 토큰 65K + 복구 폴백(이미 구현)이 대용량 흐름 트리의 전제.
- **대형 문서 대응(미결 #4)**: 1패스 추출은 차원 트레이드오프(PoC: D↑ C↓)가 있음. 화면 수가 많으면 **화면 단위 분할 추출 → 병합** 파이프라인 고려.

### 6.2 DB / 모델
- `document.menu_tree`(TEXT)에 흐름 트리 JSON 저장 — **컬럼 변경 없음**, JSON 구조만 변경. `format` 필드로 구/신 구분.
- 상태 매트릭스(`state_inventory`)는 **PR 노드 생성의 입력**으로 활용 (별도 추출 → PR 분기 보강). 또는 흐름 추출에 흡수.

### 6.3 TreeViewPage (프론트)
- 흐름 트리 렌더링: 타입 배지(PR/C/D…) + 들여쓰기 트리. 현재 노드 카드 UI 확장.
- 편집: 노드 추가/삭제/이동, 타입·content·spec_page 수정. (현재는 제외 토글만 → 편집 기능 확장 필요)
- 좌→우 그리드 미리보기는 Excel로 위임 가능.

### 6.4 Excel export
- 흐름 트리 그리드: **PoC 렌더러를 정식화** (범례 + Step 헤더 + 타입색 + 트라이 펼침). [documents.py](../backend/app/api/documents.py)의 export 교체/추가.
- 기존 메뉴트리 export는 구 포맷 호환용으로 유지하거나 폐기.

### 6.5 TC 파생 (신규) — `linearize_flow_tree(tree) -> [tc]`
- 각 루트→종단 경로를 순회하며 TC 생성:
  - 경로의 PR content → `preconditions`
  - C/T/H/DC/RC → `steps[].action` (+ 직후 D → `steps[].expected`)
  - 마지막 D/V → `expected_result`
  - `category` = 루트 menu_path, `spec_page` = 종단 노드 상속, `tc_type`/`priority`는 휴리스틱 또는 경량 분류
- 결과를 기존 `testcases`에 저장 → 검토·리포트 파이프라인 그대로.

### 6.6 추적성 / 피드백 연계
- 노드 `id`가 피드백·TC 매핑의 안정 키 (직전 category=경로 매핑을 id로 승격).
- [feedback_system_design.md](feedback_system_design.md)의 골든 비교가 흐름 트리 단위로 정밀해짐.

## 7. 마이그레이션 전략
- **하위호환**: `format` 필드로 구(structural)/신(flow) 트리 동시 지원. 렌더러·export가 분기 처리.
- **기존 프로젝트**: 자동 변환 불가(정보량이 다름) → 흐름 트리가 필요하면 **재추출**. 구 트리는 그대로 조회 가능.
- **신규 프로젝트**: 흐름 추출이 기본.
- DB 마이그레이션 없음(JSON 구조 변경뿐). TC 테이블·리포트 호환.

## 8. 단계별 구현 계획

| 단계 | 내용 | 비고 |
|---|---|---|
| **0. 태깅 관례 합의** | QA팀과 §4 확정 (입력=T/D, V 분리, 중복표기) | 선행 필수 — PoC 3종 보여주고 결정 |
| **1. 추출** | `FLOW_PROMPT` 정식화 + §3 스키마 출력, `format:"flow"` 저장 | PoC v3 기반 |
| **2. Excel export** | PoC 그리드 렌더러 정식화 (구/신 분기) | 위험 낮음 |
| **3. TC 선형화** | `linearize_flow_tree` 구현 → 기존 testcases 적재 | 기존 리포트·검토 재사용 |
| **4. TreeViewPage** | 흐름 트리 렌더 + 편집 UI | 프론트 작업량 큼 |
| **5. 대형문서 분할추출** | 화면 단위 추출→병합 (필요 시) | 1패스 트레이드오프 해소 |
| **6. TC 레벨 재해석** | 선형화 깊이/범위 정책으로 재정의 | 미결 #3 |

> 권장 순서: **0 → 1 → 2** (추출·출력을 먼저 안정화, 사람이 Excel로 검토 가능하게) → **3**(TC 파생) → **4**(편집 UI). 1·2까지만 해도 "사람 양식 트리를 자동 생성"이라는 핵심 가치는 달성.

## 9. 리스크 & 미결 질문
1. **태깅 관례** (§4) — QA 합의 필요. 가장 먼저.
2. **1패스 추출의 차원 트레이드오프** — PoC v3에서 D 강조 시 C/T 감소 관찰. 대형 문서는 화면 단위 분할추출 검토.
3. **TC 레벨 시스템의 의미** — 흐름 트리가 전체 분해를 담으면 Lv.1~5는 "선형화 시 어느 깊이/경로까지 TC화할지" 정책으로 재정의.
4. **편집 UX** — 흐름 트리는 노드가 많고 깊어 편집 부담 큼. 일괄 편집·접기/펼치기·검색 필요.
5. **추출 비용/시간** — 흐름 트리는 출력이 커 토큰·지연 증가. 진행률 표시·분할 추출로 완화.
6. **tc_type/priority 자동 분류** — 선형화 시 휴리스틱 vs 경량 AI 분류.
