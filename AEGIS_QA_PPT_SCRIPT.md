## [SLIDE 2] 핵심 메시지 (Executive Summary)

**헤드라인:**
> "기획서(PDF) 한 장으로 수백 개의 테스트케이스를 자동 생성하는
> AI QA 시스템을 사내에 구축했습니다."

**3줄 요약:**
- 기획서 PDF를 업로드하면 AI가 화면을 분석하고 TC를 자동 생성
- XpERP뿐 아니라 **사내 모든 웹 서비스**에 적용 가능한 범용 구조
- TC 생성 → 검토 → 수정 → Excel 리포트 → Playwright 자동화까지 **전 주기 지원**

## [SLIDE 3] 기존 QA 프로세스의 문제

**현황 (As-Is):**

| 단계 | 현재 방식 | 문제점 |
|------|-----------|--------|
| TC 작성 | 담당자가 수동으로 작성 | 시간 과다, 누락 빈번 |
| TC 품질 | 작성자 역량에 의존 | 커버리지 불균형 |
| TC 유형 | 정상 케이스 위주 | 비정상/경계값/예외 누락 |
| 변경 대응 | 기획 변경 시 전면 재작성 | 유지보수 부담 |
| 자동화 | 대부분 수동 실행 | 반복 비용 지속 발생 |

**결론:** 기능이 늘어날수록 QA 비용이 선형 증가하는 구조

## [SLIDE 4] 솔루션 개요 — Aegis QA Assistant

**한 줄 정의:**
> PDF 기획서 + AI = 완성된 테스트케이스

**핵심 플로우:**

```
[기획서 PDF]
    ↓ 업로드
[Claude Vision]  ← 페이지 이미지 분석
    ↓
[기능 목록 추출]
    ↓
[TC 자동 생성]  ← RAG로 웹 UI 지식 보강
    ↓
[검토 / 수정]   ← 담당자 리뷰
    ↓
[Excel 리포트]  + [Playwright 자동화 스크립트]
```

**유형 자동 분류:**
- 정상(Positive) 40% / 비정상(Negative) 30% / 경계값(Boundary) 20% / 예외(Exception) 10%

## [SLIDE 6] 핵심 기능 ① — AI 기반 TC 자동 생성

**동작 원리:**
1. PDF 업로드 → PyMuPDF로 페이지를 이미지로 변환
2. Claude Vision API가 각 페이지 이미지를 분석 → 기능 목록 추출
3. 추출된 기능별로 Claude가 TC 자동 생성

**생성되는 TC 필드:**

| 필드 | 내용 |
|------|------|
| TC ID | 자동 채번 (TC-001, TC-002 ...) |
| 카테고리 | 기능 영역 자동 분류 |
| 제목 | 테스트 목적 한 줄 요약 |
| 전제 조건 | 테스트 실행 전 준비 사항 |
| 테스트 단계 | 구체적 실행 절차 (Step-by-Step) |
| 기대 결과 | 정상 동작 판정 기준 |
| 유형 | 정상 / 비정상 / 경계값 / 예외 |
| 우선순위 | High / Medium / Low |

**포인트:** 기획서 한 장당 평균 수십 개 TC 자동 생성 (기능 복잡도에 따라 다름)

## [SLIDE 7] 핵심 기능 ② — 웹 UI 지식 자동 수집 (크롤러)

**왜 필요한가:**
> 기획서 PDF만으로는 실제 화면의 버튼명·입력 필드·컬럼명을 알 수 없음
> → 실제 서비스 화면을 크롤링해서 AI 지식으로 축적

**동작 방식:**
1. Playwright로 대상 서비스에 자동 로그인
2. 사이드바 메뉴 전체 파싱 (예: XpERP 325개 메뉴)
3. 각 메뉴 화면 이동 → 스크린샷 캡처
4. **Gemini Vision**이 스크린샷 분석 → 화면 구성요소 텍스트 추출
5. Gemini Embedding으로 벡터화 → 벡터 DB 저장

**수집되는 정보 예시 (단지정보등록 화면):**
```
화면 제목: 단지정보등록
검색 조건: 단지명, 단지코드
입력 필드: 단지명(텍스트), 단지형태(드롭다운), 계약시작일(날짜)
           회계일마감(라디오), 전액마이너스(체크박스) ...
버튼: 조회, 신규, 저장, 새로고침
그리드 컬럼: 단지명, 단지코드, 전산업체 ...
```

**확장성:** 대상 URL과 로그인 정보만 변경하면 어떤 웹 서비스든 적용 가능

## [SLIDE 8] 핵심 기능 ③ — RAG 기반 TC 품질 향상

**RAG (Retrieval-Augmented Generation)란:**
> TC 생성 시 벡터 DB에서 관련 화면 정보를 실시간으로 검색해
> AI 컨텍스트에 주입 → 실제 화면에 맞는 정확한 TC 생성

**RAG 적용 전/후 비교:**

| 항목 | RAG 미적용 | RAG 적용 |
|------|-----------|---------|
| 버튼명 | "저장 버튼 클릭" (추측) | "저장 버튼 클릭" (실제 확인) |
| 필드명 | "이름 입력" (일반적) | "단지명 입력 (최대 50자)" (정확) |
| 경계값 | 임의 설정 | 실제 화면 제약 기반 |
| TC 정확도 | 개발자 수정 필요 | 즉시 활용 가능 수준 |

**지식 구조:**

```
벡터 DB
├── manual_xperp (PDF 매뉴얼 청크)
└── web_ui_xperp (실제 화면 정보)
    ├── 단지관리 > 단지정보 > 단지정보등록
    ├── 단지관리 > 단지정보 > 동호생성
    └── ... (325개 화면)
```

## [SLIDE 9] 핵심 기능 ④ — TC 검토 및 관리 워크플로우

**생성 이후의 흐름:**

```
[TC 자동 생성]
    ↓
[담당자 리뷰]

         ↓
    [AI 재생성] ← 수정 피드백 반영
         ↓

    ↓
[Excel 리포트 다운로드]
```

**관리 기능:**
- TC 유형별 / 우선순위별 / 검토상태별 필터링
- TC 개별 직접 수정 (제목, 단계, 기대결과)
- 변경 이력 추적 (ChangeType: Added / Modified / Deleted)
- 문서별·프로젝트별 통계 대시보드

## [SLIDE 10] 핵심 기능 ⑤ — Playwright 자동화 스크립트 생성

**기능 설명:**
> 검토 완료된 TC를 기반으로 Playwright Python 테스트 스크립트 자동 생성
> → 사람이 직접 실행하던 TC를 코드로 자동화

**생성 결과물 (Python .py 파일):**
```python
# TC-001: 단지정보 등록 - 정상 케이스
async def test_단지정보_정상_등록(page):
    # 전제 조건: 로그인 완료 상태
    await page.goto("https://xperp.co.kr/main.do")
    # Step 1: 단지관리 > 단지정보 > 단지정보등록 메뉴 클릭
    await page.click("text=단지관리")
    ...
    # 기대 결과: "저장되었습니다" 메시지 확인
    await expect(page.locator(".toast")).to_have_text("저장되었습니다")
```

**의의:**
- TC 작성 → 자동화 코드 생성까지 **원스톱 처리**
- 반복 회귀 테스트 자동 실행 가능
- CI/CD 파이프라인 연동 기반 마련

## [SLIDE 11] 확장성 — 사내 모든 웹 서비스 적용 가능

**현재 구성 (XpERP 특화):**

```
XpERP 로그인 → 메뉴 파싱 → 화면 크롤링 → 벡터 DB
```

**범용 확장 구조:**

```
[서비스 A] ─┐
[서비스 B] ─┼─→ Aegis QA Assistant ─→ 서비스별 벡터 DB
[서비스 C] ─┘              ↑
                   PDF 매뉴얼 + 크롤러
```

**적용 가능 서비스 유형:**
- 사내 관리 시스템 (ERP, 그룹웨어, 인사 시스템 등)
- 고객 대상 웹 서비스 (B2B SaaS, 포털 등)
- 개발 중인 신규 서비스 (기획서만 있어도 즉시 TC 생성 가능)

**확장 시 필요한 것:**
1. 대상 서비스 URL + 로그인 계정
2. PDF 형태의 기획서 또는 매뉴얼
3. (선택) 크롤러 로그인 커스터마이징 (OAuth, OTP 등 지원)

**결론:** 서비스마다 새로 만들 필요 없음. 설정값만 추가하면 동일 시스템 재사용
